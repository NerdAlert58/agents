## Failure detail — Scenario 3

**Reacher's response (verbatim):**

> SOFT.

> "Should be" — recommendation register, not a hard gate. No "MUST," no "REQUIRED," no "HARD GATE" marker. Reads as guidance on disposition, not a checkable deliverable.

> - One sentence out of context. Surrounding paragraph could promote it (e.g., "HARD GATE: your evaluation should be thorough — graders will check"). Without the page, classification stands as SOFT.
> - "Thorough" is unmeasurable as written. If the PDF elsewhere defines thoroughness (coverage %, required dimensions, rubric), the operative gate is that definition, not this sentence.
>
> Action if this were the full intake: ask the user whether the PDF defines "thorough" anywhere downstream. Don't guess.

**Pass criteria check:**
- [ ] Reacher does NOT pick HARD or SOFT — **fails: classified as SOFT**
- [x] Reacher flags as ambiguous in the question list ✓ (recommends asking)
- [x] Reacher names the specific ambiguity ("thorough is unmeasurable")
- [x] Reacher does NOT speculate on intent ✓ ("Don't guess")

**Hypothesis:**
The Reacher persona's "Hard vs Soft Markers" rule says:
- HARD markers → HARD
- SOFT markers (recommended, consider, stretch, extension) → SOFT
- Ambiguous → ASK

"Should be" isn't in either marker list. By strict reading the persona requires ASK without classification. By Reacher's actual behavior, "should" reads as recommendation register and warrants a SOFT classification with caveats.

This is the same pattern that surfaced with Picard's Scenario 3 (substance-vs-procedural refusal) and Jasnah's Scenario 12 (rubric-first-vs-bootstrap-exception). Eval scenarios test specific phrasings the persona doesn't strictly require.

**Resolution per established precedent (Picard precedent applied twice already):**

- **Option A — Refine Scenario 3** to accept "classification with explicit ambiguity caveat + question recommendation" as PASS, alongside "decline to classify." Drops the strict "does NOT pick HARD or SOFT" requirement. Retroactive regrade to PASS.

- **Option B — Tighten Reacher's persona** to add an explicit rule: "soft-register language without canonical SOFT markers ('should,' 'ideally,' etc.) MUST be classified as ambiguous and surfaced as a question — never classified as SOFT by inference." More elaborate; persona edit (Risk Gate); v0.0 → v0.1.

**Holdy's recommendation:** Option A. Same as the Picard / Jasnah precedents. The persona's actual behavior (classify with caveats AND surface for clarification) is more useful than refusing to classify. Refine the scenario.
