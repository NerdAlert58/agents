>    Supervisor hands raw multimodal payload to a worker whose only legitimate output is structured extractions (entities, observations, provenance offsets). > 3. **Supervisor → Evidence-Retriever (Boundary C: trust-internal, read-only egress).**
>    Worker queries evidence corpus / FHIR / guideline store. Enforce: read-only credentials, query templates not free-form SQL, patient-scope filter injected by supervisor not worker. Failure mode: worker is prompt-injected by extracted text and exfiltrates broader patient context — mitigation: supervisor strips untrusted text before passing query intent; retriever never sees raw extracted strings as instructions, only as parameters.
>
> 4. **Workers → External services (Boundary D: egress).**
>    Multimodal model endpoint and evidence store sit outside the agent trust zone. Treat responses as untrusted input on return. Enforce: TLS, per-worker credentials (no shared secret), rate-limit budgets, response schema validation. Failure mode: upstream returns malformed or malicious content — schema-validate or discard.

> 5. **Workers → Supervisor (Boundary E: return path).**
>    Worker outputs are untrusted until validated. Supervisor re-checks: provenance present, citations resolvable, patient-scope unchanged, no instructions embedded in fields. ## Scenario 5 — explicit-boundary-enforcement

> The boundary itself is not self-enforcing. Retrieved chunks are just strings; once they enter the prompt, the model treats them as instructions unless something upstream prevents that. So the enforcement is layered, and each layer is a decision you have to defend.
>
> **1. Provenance tagging at retrieval time.** Wrap every retrieved chunk in a structured envelope — XML tags, JSON fields, whatever — that the system prompt declares as untrusted data. Example: `<retrieved_document source="..." trust="data-only">...</retrieved_document>`. The system prompt instructs the model: content inside these tags is reference material, never instructions. This is necessary but weak on its own — prompt injection defeats it routinely.
>
> **2. Content sanitization before insertion.** Strip or escape known injection vectors: instruction-shaped strings ("ignore previous," "system:"), embedded tool-call syntax, zero-width characters, role markers. For clinical content, also strip anything matching the response schema you expect downstream, so a poisoned document can't pre-fabricate an answer.
>
> **3. Output-side constraints, not input-side trust.** The honest enforcement lives after the LLM, not before. Constrain the model to a structured output schema (tool call, JSON schema, citations-required format). If the answer must include citations to retrieved doc IDs, an injection that produces an uncited claim fails validation. This converts "did the model obey injection" into a detectable schema violation.
>
> **4. Allowlist of action surfaces.** The model's output cannot trigger writes, tool calls, or external effects directly from retrieved-content reasoning. Any state-changing action requires a separate, narrower call with its own input validation. The retrieval-augmented turn is read-only by construction.
>
> **5. Audit logging of the boundary crossing.** Every retrieved chunk that enters context gets logged with hash, source, and retrieval query. When something goes wrong, you can prove which document influenced which response.
>
> **The hostile question:** "How do you stop a poisoned EHR note from making the model recommend the wrong drug?" Answer: provenance tags reduce probability, sanitization catches known patterns, schema-constrained output forces citations, and the action allowlist means the model can't write back to the chart unprompted. No single layer is sufficient. Defense is the stack.
>
> Assumption flagged: this presumes the LLM provider exposes structured-output enforcement. If not, layer 3 weakens significantly and you compensate with a second validator model — expensive, and worth naming as a trade-off in ARCHITECTURE.md.

## Scenario 6 — failure-modes-before-happy-path

> When a citation can't be resolved, the verification layer must fail closed — never silently. The contract:
>
> 1. **Block the answer.** The unverifiable claim does not reach the user as authoritative output. A clinical tool that fabricates a source is worse than no tool.
>
> 2. **Classify the failure.** Three distinct modes, each handled differently:
>    - *Source unreachable* (FHIR endpoint timeout, doc store 5xx) — retryable, transient. Surface as "verification pending," queue retry, log to observability.
>    - *Source resolved but content drifted* (citation points to a doc whose hash no longer matches what was indexed) — staleness. Quarantine the answer, flag the index for refresh.
>    - *Source does not exist* (hallucinated citation, broken ID) — model failure. Reject the answer entirely, log as a hallucination event, increment the per-model trust counter.
>
> 3. **Degrade visibly.** Return to the caller with an explicit "unverified — citation [X] could not be resolved [reason]." The user sees the gap; the upstream agent decides whether to retry, reroute, or abandon.
>
> 4. **Emit telemetry.** Every unresolved citation is an event with: citation ID, failure class, model, query, timestamp. This feeds the audit trail and the trust dashboard. Without this, you cannot defend the system to a CTO asking "how often does it lie?"
>
> 5. **Never auto-substitute.** The verification layer does not pick a "close enough" source. Substitution is a clinician decision, not a system decision. Crossing that boundary breaks the trust contract.
>
> The hostile interview question this answers: *"What stops your system from inventing a citation and shipping it as fact?"* — Answer: the verification layer treats unresolved as failure, not as friction to route around.
>
> Assumption: the citation registry exposes a deterministic resolve(id) -> {hit, hash, content} contract. If it doesn't, that's a prerequisite, not a verification-layer concern — flag back to the retrieval design.

## Scenario 9 — assumptions-marked

> Anorak here.
>
> 1000 req/min is 16.6 req/sec sustained. Treat that as the hard ceiling, not a target. Design for ~60% steady-state utilization (10 rps) so bursts don't trip the limit.
>
> **Budget allocation (assumption — flag in ARCHITECTURE.md, verify against actual FHIR vendor SLA):**
> - 70% — interactive user reads (chart loads, search)
> - 20% — background sync / cache warm
> - 10% — write-backs and headroom
>
> At 300 concurrent users that's ~2 req/user/min average. Workflows that fan out (e.g., a patient summary pulling Patient + Encounter + Observation + MedicationRequest) can blow the budget in one click. Mitigations, in order of preference:
>
> 1. **Read-through cache with TTL per resource type.** Patient demographics: 1hr. Observations: 60s. MedicationRequest: 30s. Cuts steady-state by 5-10x. Trade-off: staleness — document per-resource TTL rationale and the clinical safety question it implies.
> 2. **Request coalescing.** Two users opening the same chart within the TTL window share one upstream call. Single-flight pattern at the gateway.
> 3. **Bulk FHIR ($export) for cohort-level work.** Anything touching >10 patients goes async via bulk, not per-resource GETs. Keeps interactive budget intact.
> 4. **Token-bucket rate limiter at the egress proxy.** 14 rps refill, 50-token burst. Hard backpressure to callers — never let the app discover the limit by getting 429'd by the vendor.
> 5. **Priority queue on the limiter.** Interactive > sync > write-back. Background sync yields under load.
>
> **Failure modes:**
> - 429 from FHIR — exponential backoff with jitter, surface "data may be stale" banner, never silently retry-storm.
> - FHIR down — serve cache with explicit staleness timestamp; block writes; alert.
> - Cache poisoning / staleness — every cached value carries source timestamp; clinician-visible.
>
> **Scale story:** at 100 users, cache is over-engineering — note it. At 1K, cache + coalescing essential. At 10K, you've outgrown a single FHIR endpoint; architecture must shard by tenant or negotiate higher limit. Say so explicitly.
>
> **Hostile question to pre-answer:** "What happens when a clinician sees a stale lab value because of your cache?" Answer in the doc, not at interview.
